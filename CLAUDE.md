# Stable Diffusion Personas

## Purpose

When OpenAI trained the CLIP model on ~400 million image-caption pairs, later U-net/diffusion-based models (in this case, Stable Diffusion 1.5 and various enthusiast public checkpoints) were able to provide a rich vocabulary and even a context-aware (attention-based) language model for image-generation. Part of this vocabulary includes hundreds of named likenesses, from images of specific people in stock photos. Combining the two abilities (vocabulary and multiple-token attention sequences) allows a simple madlib generator to create endless stable, unique faces, disimilar to any living person—using a simple schema such as "caucasian model, Jane Doe, Janet Doe, Jenny Doe" with three to seven celebrity names from a long list that have been manually tested to be in the model. Batching and curating the thousands (10^150+ because order matters, = 100! or much higher) of possible output faces creates a library of dozens or hundreds of high-quality reusable "persona" prompts, about 100 to 150 characters of text. This means that the model already contains the ability to generate an infinitude of stable, reusable faces, it's just not exposed to the prompt-writing user in any meaningful fashion, short of the brute-force madlib prompt-script-generating approach just mentioned. Downsides of this approach include (1) the found, long prompts leave too little room for more tokens of context to do anything interesting with the face, especially other parts of the image, and (2) having to track and curate all of this oneself, and having to run the script separately to get new faces. (Also the prompt includes the names used to achieve the face, which is like unobfuscated source code in a way.)

## Experimental Approaches for the Personas Project

### A. Pseudo-token File

One better approach would be something pseudo-token based, where the user could use the prompt (directly, possibly avoiding writing out images) with a single model/checkpoint to train and bake-out a small file (768 or maybe 3,072 scalars) that could be reused in a prompt, like so: "`soft studio lighting, fantasy style painting, <pref:dbuk>, riding a dragon, smiling`" and the appropriate file "`dbuk.bin`" or whatever would be loaded from disk and injected into the token stream / embedding, effectively extending the vocabulary of the CLIP transformer. This milestone is important to prove that the pseudo-tokens can be used in the way normal text strings can be used: entirely flexibly, in terms of context or attention. Prior art in this area includes image-based training approaches, including Textual Inversion (pretty similar, fairly lightweight), LoRA (robust, pretty standard for end-users), and Dreambooth (a bit overkill but amazing for capturing an entire style or something richer or more complex). This pseudo-token approach most closely resembles Textual Inversion, with some distinctions (mainly not having to create images).

Another capability would be to allow blending or averaging (linear combination, if possible) between two or more pseudo-tokens, because we have seen from a basic test that this does seem to work on some level, but this needs to be explored more.

### B. "pref" System, Inspired By Midjourney's Style "sref" system

The end goal would be to would train a pair of slightly larger models so the user can just write `<fpref=#>` or `<mpref=#>` to get female or male faces by simply picking a single random 64-bit integer—or writing `<fpref=?>` which gets expanded to a random integer for the user, but the actual number is written out with the image file, along with the full prompt, and seed and other parameters, so the user can curate their own set of face integers. To get this off the ground, this project will collect up two folders with a library of hundreds or thousands of curated pseudo-token files. The first step is to use the 64-bit integer as a seed in a PRNG that initializes a reproducible noise vector of 32 scalars, then train a network (including a simple MLP) that can map the noise vector to a pseudo-token vector of width 1 to 4 tokens (768 to 3,072 scalars) that is hopefully "on the manifold" of aesthetically pleasing faces. The sort of existence proof of such a low Kolmogorov complexity is that the faces on the manifold are already in the model and the prompts to get them only use around 100 to 150 Unicode characters each.

The final capability would be to allow more control of faces, similar to the blending feature mentioned in A above, while keeping the textual prompt writing format very simple from the user's perspective.

### C. Back to Swift

For now this exploration will be done with Python SD, diffusers, PyTorch, MPS device (on Mac with M1 Pro chip, for training and inference). If possible, it would be good to bring this back to CoreML to run with Mochi Diffusion and other Swift-based SD implementations, converting the models, injecting the pseudo-token, etc. It probably doesn't need to run on iPad, just the Mac.
